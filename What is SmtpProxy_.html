Why was SmtpProxy created?

Both GMail and Microsoft Live Mail provide SMTP servers for sending email, and they both require an encrypted (SSL/TLS) connection. It is good that they do this, but it makes it difficult to use email client software that does not support SSL/TLS.

When my company ran into this problem, we set up an internal SMTP server and had it relay the mail to GMail/Live, but this led to other problems. Some software uses the same user credentials for logging into the SMTP server AND for sending the email. So, email that should have had a reply to email address of "support@mycompany.com" ended up with a reply to of "smtpuser@mycompany.internal". To work around this, we created SmtpProxy.

What does SmtpProxy do?

SmtpProxy listens on a port you choose; the default port is 25. When SmtpProxy detects a connection on this port, it does the following:

  • Connects to the SMTP server you configure (GMail / Microsoft Live Mail / any other SMTP server)
  • Starts an encrypted connection with the SMTP server
  • Returns the connection information to your email client


After the inital connection is established, SmtpProxy acts as a middleman. Any data received from your email client is forwarded to the SMTP server and any data returned from the SMTP server is sent to your client.

SmptProxy runs as a Windows service so it can be installed on a LAN server and used by your entire office. Note that SmtpProxy does NOT login to the SMTP server, so each user on the LAN can use their own credentials.

SmtpProxy can also be installed on a standalone Windows machine. It uses very little memory and almost no CPU.
